# The Execute Contract

The `execute` function is where your tool actually does work. Its signature looks small but encodes important contracts.

## The signature

```ts
execute: (
  toolCallId: string,
  args: ValidatedArgs,
  signal?: AbortSignal,
  onUpdate?: (partial: ToolResult) => void,
) => Promise<ToolResult>;
```

Four parameters, one return type. Each one matters.

### Parameter 1: `toolCallId`

The ID of this tool invocation, generated by the LLM. Useful for:

- Logging ("tool call abc123 took 4.2s")
- Correlating with the assistant message that requested it
- Updating UI state keyed by ID
- Some tools (like long-running shells) keep state per ID

You usually don't *need* it for basic tools. Pass through if you do.

### Parameter 2: `args`

The validated arguments. By "validated" we mean: your schema check (Lesson 3.2) passed, so the args conform to the schema.

This means inside `execute`, you can trust the types. No defensive checks needed:

```ts
async execute(_id, args: { path: string }) {
  // args.path is definitely a string here
  const content = await fs.readFile(args.path);
}
```

### Parameter 3: `signal`

An `AbortSignal` that fires when:

- The user hits `Esc` in the UI
- The agent times out
- The agent is destroyed

Your tool **must honor this signal**, especially for long operations:

```ts
async execute(_id, args, signal) {
  signal?.throwIfAborted();  // bail out early if already aborted

  for await (const chunk of someLongStream(args.input)) {
    signal?.throwIfAborted();  // check periodically
    process(chunk);
  }
}
```

For network requests, pass it through:

```ts
async execute(_id, args, signal) {
  const response = await fetch(args.url, { signal });
  return ...;
}
```

For child processes, kill them on abort:

```ts
async execute(_id, args, signal) {
  const child = spawn("long-running-command");
  signal?.addEventListener("abort", () => child.kill("SIGTERM"));
  // ...
}
```

**A tool that ignores `signal` is a tool that hangs your agent.** Treat it as required.

### Parameter 4: `onUpdate`

A callback you call to stream partial results. Useful for:

- Long-running tools showing progress ("processed 12 of 100 files")
- Tools that produce streaming output (a `bash` command that prints as it runs)
- Tools that have a "checkpoint" they want to surface mid-execution

Example:

```ts
async execute(_id, args, signal, onUpdate) {
  const files = await listFiles(args.dir);
  for (let i = 0; i < files.length; i++) {
    signal?.throwIfAborted();
    await processFile(files[i]);
    onUpdate?.({
      content: [{ type: "text", text: `Processed ${i + 1}/${files.length}` }],
      details: { progress: (i + 1) / files.length },
    });
  }
  return { content: [{ type: "text", text: "Done" }], details: { count: files.length } };
}
```

The harness will emit `tool_execution_update` events for each `onUpdate` call. The UI can render a progress bar or live-updating text.

This is optional. Most tools don't need it. Use when you have it.

### Return type: `ToolResult`

```ts
interface ToolResult<T = unknown> {
  content: (TextContent | ImageContent)[];
  details?: T;
  terminate?: boolean;
}
```

Three fields:

#### `content` — what the LLM sees

This goes into the `tool_result` block in the next user message. Text and images supported (most providers).

```ts
return {
  content: [{ type: "text", text: "File contents: ..." }],
  details: { ... },
};
```

For multi-modal tools (e.g. `read` on an image file):

```ts
return {
  content: [
    { type: "text", text: "Image: 1024x768 PNG" },
    { type: "image", data: base64, mimeType: "image/png" },
  ],
  details: { ... },
};
```

#### `details` — what the UI uses

Arbitrary structured data. Doesn't go to the LLM. Used by your UI components for rendering hints (truncation info, file size, exit code, etc.).

```ts
return {
  content: [{ type: "text", text: output }],
  details: {
    exitCode: 0,
    duration: 1234,
    truncated: false,
  },
};
```

In Chapter 8 you'll see the UI rendering the `bash` tool's `details.exitCode` to show a colored badge.

#### `terminate` — early-stop hint

If `true`, hint to the agent that the loop should stop after this batch. Useful for tools like `attempt_completion` (the agent declares "I'm done").

```ts
return {
  content: [{ type: "text", text: "Task complete." }],
  details: {},
  terminate: true,
};
```

Subtle: the agent only stops if **every** tool in the batch returns `terminate: true`. One terminating tool can't unilaterally halt others. We'll see this in Chapter 4.

## Errors: throw, don't return

Wrong:

```ts
async execute(_id, args) {
  if (!fs.existsSync(args.path)) {
    return {
      content: [{ type: "text", text: "File not found" }],
      details: {},
    };
  }
}
```

Right:

```ts
async execute(_id, args) {
  if (!fs.existsSync(args.path)) {
    throw new Error(`File not found: ${args.path}`);
  }
}
```

The harness catches the throw, builds a `tool_result` with `is_error: true`, and the LLM sees the error message in context. The LLM will usually try a different approach next turn.

If you encode errors as content with `is_error: false`, the LLM might think the call succeeded with a weird response. Don't.

## A complete example

Let's write `bash` end-to-end:

```ts
import { Type, Static } from "typebox";
import { spawn } from "node:child_process";

const bashSchema = Type.Object({
  command: Type.String({
    description: "Bash command to execute. Will be run via `bash -c`.",
  }),
  cwd: Type.Optional(Type.String({
    description: "Working directory. Defaults to the agent's cwd.",
  })),
});

export const bashTool = {
  name: "bash",
  description: "Run a bash command. Returns combined stdout+stderr and the exit code. Output is truncated to 10KB if longer. Long-running commands can be aborted.",
  parameters: bashSchema,
  async execute(
    _id: string,
    args: Static<typeof bashSchema>,
    signal?: AbortSignal,
    onUpdate?: (partial: any) => void,
  ) {
    return await new Promise<{ content: any[]; details: any }>((resolve, reject) => {
      const child = spawn("bash", ["-c", args.command], {
        cwd: args.cwd ?? process.cwd(),
      });

      let buffer = "";
      const MAX = 10_000;

      const append = (data: Buffer) => {
        buffer += data.toString("utf-8");
        if (buffer.length > MAX) {
          buffer = buffer.slice(0, MAX) + "\n[... truncated ...]";
        }
        onUpdate?.({
          content: [{ type: "text", text: buffer }],
          details: { running: true },
        });
      };

      child.stdout.on("data", append);
      child.stderr.on("data", append);

      const onAbort = () => {
        child.kill("SIGTERM");
        reject(new Error("Aborted"));
      };
      signal?.addEventListener("abort", onAbort);

      child.on("exit", (code, sig) => {
        signal?.removeEventListener("abort", onAbort);
        if (signal?.aborted) return;

        const exitInfo = sig ? `[killed: ${sig}]` : `[exit ${code}]`;
        const text = buffer ? `${buffer}\n${exitInfo}` : exitInfo;

        if (code !== 0) {
          // Non-zero exit is NOT a thrown error — it's a tool result the LLM should see
          resolve({
            content: [{ type: "text", text }],
            details: { exitCode: code, signal: sig, truncated: buffer.length >= MAX },
          });
        } else {
          resolve({
            content: [{ type: "text", text }],
            details: { exitCode: 0, truncated: buffer.length >= MAX },
          });
        }
      });

      child.on("error", err => {
        signal?.removeEventListener("abort", onAbort);
        reject(err);
      });
    });
  },
};
```

What this demonstrates:

- **Signal honored**: `SIGTERM` on abort, abort listener cleaned up
- **Streaming output**: `onUpdate` called as data arrives
- **Truncation**: 10KB cap with marker
- **Exit codes**: returned in `details`, not as errors. The LLM sees command failures and decides how to respond.
- **Spawn errors are thrown**: `spawn` itself failing (binary not found) is an error, not a result

Compare to pi's production `bash.ts` (`packages/coding-agent/src/core/tools/bash.ts`) — it adds: timeout, foreground/background tracking, child process registry for cleanup on agent shutdown, output operation parsing for special markers. Same shape, more features.

## Common patterns

### Reading remote data

```ts
async execute(_id, args, signal) {
  const response = await fetch(args.url, { signal });
  if (!response.ok) {
    throw new Error(`HTTP ${response.status}: ${response.statusText}`);
  }
  const text = await response.text();
  return { content: [{ type: "text", text }], details: { url: args.url } };
}
```

### Atomic file writes

```ts
async execute(_id, args, signal) {
  signal?.throwIfAborted();
  const tmp = `${args.path}.tmp.${Date.now()}`;
  await fs.writeFile(tmp, args.content);
  await fs.rename(tmp, args.path);
  return {
    content: [{ type: "text", text: `Wrote ${args.content.length} bytes to ${args.path}` }],
    details: { bytes: args.content.length },
  };
}
```

### Progress on long operations

```ts
async execute(_id, args, signal, onUpdate) {
  const items = await loadItems();
  const results = [];
  for (let i = 0; i < items.length; i++) {
    signal?.throwIfAborted();
    const result = await processItem(items[i]);
    results.push(result);
    onUpdate?.({
      content: [{ type: "text", text: `Processed ${i + 1}/${items.length}` }],
      details: { progress: (i + 1) / items.length },
    });
  }
  return { content: [{ type: "text", text: `Done. ${results.length} items.` }], details: { results } };
}
```

## Gotchas

- **Don't block the event loop.** Sync `fs.readFileSync` on a 100MB file freezes everything, including the abort handler. Always use async APIs for long ops.
- **Don't catch `signal.aborted` errors and rethrow them as different errors.** Let the abort propagate cleanly so the harness knows it was an abort, not a real error.
- **Don't return huge `content`.** Truncate. The LLM has a context window.
- **Don't put sensitive data in `details`.** Sessions get persisted to disk; assume `details` is logged.

## Stop and try this

Build a tool that gets the time in a given timezone and try it manually:

```ts
import { Type, Static } from "typebox";

const timeSchema = Type.Object({
  timezone: Type.Optional(Type.String({ description: "IANA timezone, e.g. 'America/New_York'" })),
});

const timeTool = {
  name: "get_time",
  description: "Get the current time in the given timezone (default: UTC).",
  parameters: timeSchema,
  async execute(_id: string, args: Static<typeof timeSchema>) {
    const tz = args.timezone ?? "UTC";
    let formatted: string;
    try {
      formatted = new Date().toLocaleString("en-US", {
        timeZone: tz,
        dateStyle: "full",
        timeStyle: "long",
      });
    } catch (err) {
      throw new Error(`Invalid timezone: ${tz}`);
    }
    return {
      content: [{ type: "text" as const, text: formatted }],
      details: { timezone: tz },
    };
  },
};

console.log(await timeTool.execute("test", { timezone: "Asia/Kolkata" }));
console.log(await timeTool.execute("test", {}));

try {
  await timeTool.execute("test", { timezone: "Not/A/Place" });
} catch (e) {
  console.log("Error caught:", (e as Error).message);
}
```

Run with `npx tsx`. Notice:

- Valid timezones return time
- Default to UTC
- Invalid throws — that's how errors propagate

## Key takeaways

1. `execute(id, args, signal, onUpdate) → Promise<ToolResult>` is the contract.
2. `signal` must be honored — call `throwIfAborted()` periodically and pass to fetches/spawns.
3. `onUpdate` streams partial results. Optional but valuable for long ops.
4. Throw on errors. Don't return error-as-content.
5. `content` is for the LLM; `details` is for the UI; `terminate` hints early-exit.
